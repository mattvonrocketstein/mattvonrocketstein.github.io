Intro

What is ymir for?

Ymir can be used to generate project boilerplate which combines fabric, boto, and puppet to create, setup, provision and control EC2 services (and, more experimentally, Elasticbeanstalk services).

After services are instantiated, it can help you with command-and-control and there is lots of functionality out of the box. Think of it as "glue code" that can help lay out project skeletons and give you a concise and relevant vocabulary for service description. It's designed to (at least theoretically) help with creation, provisioning, etc in general regardless of the instance OS, but for now at least, some parts of it might still be specific to Ubuntu servers. More specifically,

  • Boto is used for AWS automation (ex: creating EC2 instance from AMI)
  • Fabric is used for command execution on local and remote hosts, and control automation after services are created
  • Puppet normally describes how services will be provisioned (ex: installing system packages) but,
  • You can also use fabric, shell, or ansible for provisioning.

Philosophy and future development

Ymir is obviously Yet Another Devops Tool in a rapidly expanding and sometimes bewildering toolbox, which may or may not make you sigh in frustration. I built this to learn more about fabric, boto, and puppet while scratching my own itch for automation. In the end though it's become more than a learning project, and is stable enough for production use. So is it like docker or ansible? Or maybe packer? How does it compare with vagrant or cloud formation or terraform? Why doesn't it use salt or chef? OMG, I don't know! But I do strongly believe that sufficiently advanced abstractions are indistinguishable from obfuscation and I think this is a bad thing. Therefore,

  • Ymir has a specific focus on EC2 at this point, and configuring digital ocean or rackspace instances, etc, is not a priority.
  • Ymir allows you to easily use puppet in stand-alone mode, similar to how you might use ansible.. there are no puppetmaster daemons or additional servers to setup.
  • Ymir simply stores service configuration variables as json in a normal file in version control, and
  • Because a "service" is typically an instance plus an application, instance descriptions and application configuration may live side by side in service.json files if you so choose.

At some point it might make sense for ymir configuration data to "compile" to terraform or cloudformation configuration.. that option will become attractive the moment it helps me avoid additional work by leveraging prior art. I'm not dogmatically attached to the underlying mechanisms ymir uses but I am attached to the simplicity of it's interface.
