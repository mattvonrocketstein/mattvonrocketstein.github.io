Ansible through a bastion host
Date Sat 01 April 2017
Tags ansible , python , devops

The Problem

The pattern of the bastion/gateway host will probably be familiar to anyone working with servers, but getting ansible to "just work", with or without a bastion between you and the target, is surprisingly esoteric. You'd think this would be possible with one simple flag or something, but it's a little more involved than that.

Prior Art

This guide was extremely helpful for me, but didn't completely solve my use-case. The issue is that it involves modifications to static files for ansible.cfg, ssh_config, and possibly inventories. I want neither global files for these things (because I work on multiple projects in multiple accounts) nor project-local files for these things (because the ansible repo is intended to remain generic).

Especially if you're experimenting with an idea or just getting started with infrastructure as code then VPCs, bastions, and hosts behind the bastion all tend to come and go. It's just not ideal to have to keep track of these things or keep different versions of these files for different environments (prod/stage/test) during any phase of work where the resources involved are intrinsically ephemeral. The ssh_config is also a bit complicated, and when it's not setup exactly right the errors are mysterious. I wanted a general way of solving the problem rather than cargo-coding around a ssh_config from project to project with small modifications.

The Solution

What is discussed here is basically a method for running ansible playbooks or ansible modules, against a host that either is, or is not behind a bastion, by generating the files, environment variables, and command-line arguments that you need in such a way that they are available just in time, and need not be tracked in version control.

All this is done with a wrapper script for invoking ansible, which uses fabric. I'm using ansible==2.4.0 and fabric==1.13.1, but the solution presented here should work with pretty much any recent version of either library.

The code for everything is available for download here (any comments are welcome). To use this wrapper script without a bastion host, simply leave out the bastion arguments in the commands below.

Example Usage

# List available commands:
$ fab -l

# SSH into a host behind a bastion:
$ fab select.host:"privateIP" select.bastion:"publicIP" ssh

# Run ansible's ping module against a host behind a bastion:
$ fab select.host:"privateIP" select.bastion:"publicIP" ansible.ping

# Run ansible playbook against a host behind a bastion:
$ fab select.host:"privateIP" select.bastion:"publicIP" ansible.playbook:"path/to/playbook","extra_vars=for_playbook"

# Ad-hoc ansible module execution on a host behind a bastion:
$ fab select.host:"privateIP" select.bastion:"publicIP" ansible.exec-module:shell,"whoami"
